# 파주 쇼핑 상세 — UX 휴리스틱 검수

> 검수자: 시니어 UX 디자이너 관점 · 대상: `travel/shopping-paju/index.html` (컨셉 시안, 데스크탑 1380px 기준)
> 평가 프레임: Nielsen 10 휴리스틱 + UX Laws + 예약 전환 흐름 + 페이지 내 내비게이션 + 인지 부하
> 검수 제외(컨셉 단계): 링크 연결·실제 전송·실데이터 정확성·SEO·성능·다국어 완성도

---

## 종합 평점: **7.4 / 10**

**한 줄 총평:** 정보 구조와 톤은 milanobycar급 프리미엄 랜딩으로 잘 짜였고 "차량 1대 고정 요금" 핵심 메시지가 일관되게 박혀 있다. 다만 **결정에 필요한 정보(차량별 가격)가 결정 패널(예약폼)보다 한 화면 아래(Fleet)에 있어 전환의 첫 관문에서 멈칫하게 만들고**, 긴 페이지인데 섹션 점프 수단이 없으며, 인원 카운터가 요금과 무관하다는 사실이 폼 안에서 충분히 드러나지 않는다.

## 전환 흐름 워크스루 — "처음 방문하는 외국인 FIT" 시나리오

가정: 홍콩에서 온 4인 가족, 파주 아울렛 당일치기를 알아보는 중. 한국어 약간 읽지만 영어 선호. 데스크탑.

1. **히어로(25vh) 도착** — 아울렛 사진 + "신세계 파주 프리미엄 아울렛" 타이틀. 분위기 좋음. 무엇을 파는 페이지인지(=쇼퍼 예약)는 아직 불명확. 우측 예약폼이 시야에 들어와야 "예약 서비스구나"를 인지하는데, 25vh 히어로 + 브레드크럼 덕에 폼 헤더가 첫 화면 상단에 걸쳐 보이는 건 **잘 된 부분**.
2. **예약폼을 본다** — "쇼퍼 예약 / 프라이빗 기사" 배지로 서비스 성격 즉시 파악 OK. 세그먼트 기본값이 **풀데이**로 선택되어 있고 예상요금 ₩120,000(편도 라벨과 충돌, 아래 H-1 참조)... **여기서 첫 혼란**. "₩120,000부터"가 뭐의 가격인지(편도? 풀데이? 어느 차?) 폼만 봐서는 모호.
3. **"이 가격이 맞나?" 확인하려 한다** — 차량은 "Staria HiLimo 1–7인"으로 고정 표시. 우리 4인엔 맞음. 하지만 **세단/코치와 가격 비교를 폼 안에서 할 수 없다.** "차량 비교" 링크를 눌러야 Fleet로 내려감 → **결정에 필요한 정보가 결정 시점에 없음(핵심 마찰).**
4. **인원을 4로 올린다** — pax 2→4. 폼에는 아무 피드백 없음(요금 불변 = 의도된 설계지만 그 사실이 폼에 안 보임). Fleet로 내려가야 비로소 8인 이상 코치가 흐려지는 걸 봄. 4인은 문제없지만 **"인원을 바꿔도 가격이 안 변하네?"가 버그처럼 느껴질 위험.**
5. **Fleet 섹션 도달** — 세 카드에 편도/왕복/풀데이 가격이 다 있음. 매우 명확. "HiLimo로 예약" 누름 → 폼으로 스크롤 업 + 카드 선택 싱크. **이 왕복 동선 자체는 잘 설계됨.** 다만 4번 단계에서 이미 한 번 막혔던 사람만 여기까지 온다.
6. **Route / Why 6 / Sample Day / FAQ** — 신뢰를 쌓는 콘텐츠. 외국인에게 결정적인 **부가세 환급·결제수단·항공편 지연 FAQ**가 잘 들어가 있어 피크엔드의 "엔드" 직전 안심 요소로 작동. 굿.
7. **클로징 CTA** — "예약하기" + 연락 채널(WhatsApp/WeChat/LINE/Kakao). 외국인 채널 커버리지 훌륭. 피크엔드 법칙상 마지막 인상 양호.

**막히는 지점 요약:** (2)예상요금의 정체 모호 → (3)차량 비교가 폼 밖 → (4)인원 변경 무반응. 전환의 가장 앞 3스텝에 마찰이 몰려 있다. 끝부분(FAQ·CTA·채널)은 강하다.

## 발견 사항 (심각도 순)

### 🔴 High

**H-1. 예약폼 기본 상태의 라벨·금액 불일치 (가시성/오류 예방)**
- 근거: Nielsen #1 시스템 상태 가시성, #5 오류 예방
- 현재 상태: HTML 정적 마크업은 세그먼트 `풀데이`가 `.on`(469행)인데, 같은 폼의 예상요금 라벨은 "예상 요금 · **편도**"(510행), 금액은 "**120,000**"(511행)으로 박혀 있다. JS `render()`가 로드 시 풀데이(₩140,000×2.2≈₩310,000)로 덮어쓰지만, **초기 페인트~JS 실행 사이 또는 JS 비활성 시 "풀데이 선택 + 편도 라벨 + 120,000원"이라는 3중 모순 상태가 노출**된다. 정적 스크린샷(part1)에서도 풀데이 탭 활성 + 하단 금액이 보인다.
- 개선안: 정적 HTML 초기값을 **JS 최종 렌더 결과와 일치**시켜라. 세그먼트 기본을 `편도`로 두려면 510~511행을 "편도/120,000"에 맞추고 469행 `.on`을 `oneway` 버튼으로 이동. 풀데이를 기본으로 유지하려면 라벨="풀데이", 금액=계산된 풀데이가(예: 310,000)로 정적 마크업을 맞춰라. **둘 중 하나로 정합성 확보.**

**H-2. 결정 정보(차량별 비교 가격)가 결정 패널 밖에 있음 (재인 vs 회상 / 근접성)**
- 근거: Nielsen #6 회상보다 재인, 근접성 법칙, "결정에 필요한 정보가 결정 시점에 있는가"
- 현재 상태: 예약폼은 **선택된 1개 차량의 1개 금액**만 보여준다. 다른 차량이 얼마인지, 왜 HiLimo가 기본인지 폼 안에서 알 수 없고 "차량 비교" 링크로 Fleet(별도 섹션)까지 내려가야 한다. 첫 방문자는 "내 인원/예산에 어느 차가 맞나"를 폼 안에서 판단 못 함.
- 개선안: 폼의 "선택한 차량" 영역을 **3택 미니 셀렉터**로 승격(세단/밴/코치 + 각 대표가 "₩140,000~"). 풀 비교는 Fleet 유지하되, 폼에서 차량 전환·대략가 인지가 끝나도록. 최소안: "선택한 차량" 줄에 현재가만이 아니라 **"세단 ₩150k · 밴 ₩140k · 코치 ₩260k 부터"** 한 줄 요약을 넣어 회상 부담 제거.

**H-3. 인원 카운터가 요금·차량과 약하게 연결됨 (시스템 상태 가시성 / 일치성)**
- 근거: Nielsen #1, #2 시스템과 현실 일치
- 현재 상태: 폼의 탑승인원 카운터는 1~13까지 올라가지만(932행) **요금에 전혀 영향 없고**(설계 의도), 정원 초과 시 피드백은 폼이 아니라 Fleet 카드 흐림(`is-over`)으로만 나타난다(927행). 기본 차량 HiLimo는 7인까지인데 인원을 8 이상으로 올리면 **폼 안에선 아무 일도 안 일어나고**(여전히 "Staria HiLimo 1–7인" 표시 + 동일 금액), 아래로 스크롤해야 HiLimo 카드가 흐려진 걸 본다. 정원 초과 안내가 결정 시점에 없다.
- 개선안: 인원 > 선택차량 정원이면 폼 안에서 즉시 인라인 경고("8인 이상은 그룹 코치를 추천합니다 →") + 자동으로 적합 차량 제안 버튼. 동시에 "차량 1대 고정 — 인원수와 무관"(513행) 카피를 카운터 **바로 아래**로 옮겨 "왜 인원 늘려도 가격이 같지?"의 답을 무반응이 발생하는 그 자리에서 제공.

**H-4. 긴 페이지에 섹션 내비게이션·진행 표식 부재 (통제와 자유 / 인지 부하)**
- 근거: Nielsen #3 사용자 통제, Miller(작업기억), 긴 페이지 길찾기
- 현재 상태: 히어로→2컬럼→Fleet→Route→Why6→SampleDay→FAQ(8)→추천→CTA로 **9개 섹션, 풀스크롤 4분할 분량**인데 sticky 폼(데스크탑) 외에 **페이지 내 점프·현재 위치 표식·"맨 위로"가 전혀 없다.** 모바일(980px↓)에선 폼이 `position:static`(341행)이 되어 sticky 예약 진입점마저 사라져, FAQ까지 내려간 사용자가 예약하려면 길게 스크롤 업해야 한다.
- 개선안: ① 모바일에 **하단 고정 예약 바**(현재 차량+예상가+"예약" 버튼) 추가 — 어느 섹션에서든 폼/Fleet로 점프. ② 데스크탑은 현 sticky 폼 유지하되 클로징 CTA의 `#bk` 앵커처럼 **Why/FAQ 구간에도 가벼운 "예약 패널로" 텍스트 링크** 1개. ③ 선택: Fleet·Route·FAQ 정도의 가벼운 섹션 점프 칩.

### 🟡 Mid

**M-1. 제출 후 완료 상태의 피드백이 약함 (가시성 / 오류 복구)**
- 근거: Nielsen #1, #9
- 현재 상태: "예약하기" 클릭 시 폼 하단에 작은 텍스트 박스("✓ 예약이 접수되었습니다…", 524행)만 표시되고 `scrollIntoView({block:'nearest'})`(950행)로 살짝 스크롤. 버튼은 여전히 "예약하기"로 남아 **재클릭 가능**하고, 입력 검증(픽업·날짜 빈값)도 없다. 처음 사용자는 "정말 됐나?" 확신이 약하다.
- 개선안: 완료 시 ① 제출 버튼을 "✓ 접수됨 — 곧 연락드려요"로 비활성 전환, ② done 메시지를 폼 상단 또는 모달로 격상해 시선 안에 두기, ③ 컨셉 단계라도 픽업·날짜 미입력 시 인라인 검증 1줄(전환 진정성). (실제 전송은 검수 제외지만 "데모 인터랙션의 UX 설계"는 평가 대상)

**M-2. 두 개의 "예약하기" CTA가 같은 라벨, 다른 동작 (일관성)**
- 근거: Nielsen #4 일관성·표준
- 현재 상태: 폼 안 "예약하기"는 **제출**(done 표시), 클로징 CTA "예약하기"는 **`#bk`로 스크롤**(856행). 동일 라벨·동일 스타일인데 한쪽은 전송, 한쪽은 이동 → 멘탈모델 충돌.
- 개선안: 클로징 CTA를 "**예약 패널로 →**" 또는 "**지금 예약 시작**"처럼 이동 의도가 드러나는 라벨로. 제출 버튼만 "예약하기"의 종착 의미를 갖게.

**M-3. "예상 요금"의 산출 근거가 폼에 안 보임 (가시성 / 일치성)**
- 근거: Nielsen #1, #2
- 현재 상태: 폼 예상가는 `price × MODE_MUL`(round 1.7, fullday 2.2)로 계산되는데(888·899행), 사용자에겐 그냥 한 숫자만 뜬다. 공항 출발 시 Fleet note의 "+₩60,000/+₩40,000"(583행)은 폼 예상가에 **반영되지 않아**, 인천공항을 픽업에 적었어도 폼은 시내가만 보여준다 → 결제 시점 기대 불일치 소지.
- 개선안: 픽업 입력에 공항 키워드 감지 시 폼 예상가에 가산 표시("+공항 ₩60,000 포함") 하거나, 최소한 예상가 옆 ⓘ로 "시내 출발 기준 · 공항 별도"를 명시. 컨셉 단계라면 후자만으로 충분.

**M-4. Fleet 카드 가격표의 "is-best=풀데이" 강조가 모드 선택과 무관 (일치성)**
- 근거: Nielsen #2, 일관성
- 현재 상태: Fleet 카드는 항상 **풀데이 행**을 파란색·큰 글씨로 강조(`is-best`, 263행)한다. 그러나 사용자가 폼에서 "편도"를 골랐다면 카드의 강조와 사용자의 선택이 어긋나 시선이 분산된다.
- 개선안: 폼의 현재 모드(state.mode)에 맞춰 **카드의 해당 행을 강조**하도록 동기화하거나(권장), 강조를 빼고 모든 행을 동일 위계로. 풀데이를 미는 게 비즈니스 의도면 "추천" 마이크로 배지로 별도 표기.

**M-5. At a Glance 6항목과 Why 6항목·FAQ의 정보 중복 (미니멀리즘 / 인지 부하)**
- 근거: Nielsen #8 심미·미니멀, Miller
- 현재 상태: 좌컬럼 At a Glance(픽업/소요/차량/기사/포함/운영시간), Why 6카드(종일대기/고정요금/짐0/도어투도어/영어기사/경유), FAQ 8개가 **"고정요금·영어기사·포함내역·픽업·소요시간"을 반복** 노출한다. 정보 자체는 같은데 3번 등장.
- 개선안: 의도된 강화 반복이라면 OK지만, 첫 방문자 인지부하를 보면 Why 6 → 4로 압축(고정요금·종일대기·도어투도어·영어기사 정도만 차별화 핵심) 권장. At a Glance와 중복되는 "포함·소요"는 한 곳에서만.

### 🟢 Low

**L-1. 브레드크럼 "쇼핑"이 여행 허브로 연결 (일치성)**
- 현재: `홈 / 여행 / 쇼핑 / 파주`인데 "쇼핑"·"여행"이 같은 `../../travel/`로 감(397행). 컨셉상 링크 검수 제외지만 정보 위계상 "쇼핑"은 여행 하위 카테고리 인덱스를 가리켜야 자연스럽다. (구현 시 반영 메모)

**L-2. 날짜/시간 input의 과거 시점 입력 방지 부재 (오류 예방)**
- 현재: `<input type="date">`에 `min` 미설정(485행). 기본값은 내일로 잘 세팅(954행)되나 사용자가 과거 날짜를 고를 수 있다.
- 개선안: `min` 속성에 오늘(또는 내일) 주입. 운영시간 06:00–22:00이므로 time에도 가이드.

**L-3. 폼 필드 묶음에 시각적 그룹핑 약함 (근접성)**
- 현재: 날짜/시간은 한 줄로 잘 묶였으나(484행), 인원/수하물 카운터와 "선택한 차량"이 같은 면 위에 평평히 나열되어 "여정 정보 / 차량·요금"의 두 의미 그룹이 시각적으로 안 갈린다.
- 개선안: 카운터+차량 위에 미세한 간격/구분면 추가해 "언제·어디 / 무엇을 타고" 두 블록으로. (라인 금지 원칙 준수 — 여백·톤으로만)

**L-4. "차량 비교" 링크 라벨 vs 목적지 불일치 (일치성)**
- 현재: 폼 "선택한 차량"의 우측 링크가 "차량 비교"(505행)인데 클래스는 `bk__veh-change`(=변경). 누르면 Fleet로 가서 비교+선택 둘 다 됨. 라벨은 "비교"인데 행동은 "변경"에 가까움.
- 개선안: "차량 비교·변경" 또는 "다른 차량 보기"로. 사소하나 재인 명료성.
